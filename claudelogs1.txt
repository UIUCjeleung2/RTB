  ---
  How It Works

  Frontend Flow:

  1. Landing page (/): Shows Login form
  2. User enters email/password ‚Üí POST to http://localhost:5001/auth/login
  3. On success: Stores JWT token in localStorage, redirects to /dashboard
  (not yet built)
  4. Register link ‚Üí Navigate to /register page
  5. Register form ‚Üí POST to http://localhost:5001/auth/register ‚Üí Navigate
  back to login

  Backend Flow:

  1. Registration (POST /auth/register):
    - Check if user exists
    - Hash password with bcrypt (10 rounds)
    - Save to MongoDB
  2. Login (POST /auth/login):
    - Find user by email
    - Compare password with bcrypt
    - Generate JWT token (7-day expiration)
    - Return token + email to frontend
  3. Protected Routes (not yet used):
    - auth.js middleware verifies JWT from Authorization header
    - Extracts user ID from token

> could you explain what jwt is and what this jwt secret does? 

‚óè What is JWT?

  JWT stands for JSON Web Token. It's a way to securely transmit information
   between your frontend and backend as a compact, self-contained token.

  Think of it like a secure ID badge that proves who you are without having
  to show your password every time.

  ---
  How Your App Uses JWT:

  1. Login Flow:

  User logs in with email/password
           ‚Üì
  Backend verifies credentials
           ‚Üì
  Backend creates a JWT token
           ‚Üì
  Frontend stores token in localStorage
           ‚Üì
  User is now "logged in"

  2. Making Authenticated Requests:

  User tries to access dashboard
           ‚Üì
  Frontend sends request with JWT token in header
           ‚Üì
  Backend verifies JWT is valid
           ‚Üì
  Backend knows who you are without checking password again

  ---
  JWT Structure:

  A JWT looks like this:
  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6IjY3NDhiZjNhNTIxYmM4ZDQwMGEzZ
  jI3OSIsImlhdCI6MTczMjc2MjY0NiwiZXhwIjoxNzMzMzY3NDQ2fQ.XwK5Zp_8rJ3kL2mNoPqR
  sTuVwXyZ1234567890

  It has 3 parts separated by dots:

  1. Header (algorithm info)
  2. Payload (your data - user ID, expiration)
  3. Signature (proves it's authentic)

  You can decode it at https://jwt.io - try pasting one in!

  ---
  What Does JWT_SECRET Do?

  The JWT_SECRET is like a password that only your server knows. It's used
  to:

  1. Sign tokens when creating them:

  // In authController.js:40
  const token = jwt.sign(
    { id: user._id },           // Data to include
    process.env.JWT_SECRET,     // ‚Üê Secret key to sign with
    { expiresIn: "7d" }         // Expires in 7 days
  );

  This creates a signature that proves the token came from your server.

  2. Verify tokens when receiving them:

  // In middleware/auth.js:9
  const decoded = jwt.verify(
    token,
    process.env.JWT_SECRET      // ‚Üê Same secret to verify
  );

  This checks if the token is:
  - ‚úÖ Actually from your server (not forged)
  - ‚úÖ Not been tampered with
  - ‚úÖ Not expired

  ---
  Why This Matters:

  Without JWT_SECRET:
  - Anyone could create fake tokens
  - Attackers could pretend to be any user
  - No way to verify token authenticity

  With JWT_SECRET:
  - Only your server can create valid tokens
  - Tampering invalidates the signature
  - Your app is secure

  ---
  Security Best Practices:

  1. Never commit JWT_SECRET to git ‚úÖ (Your .gitignore already blocks .env)
  2. Use a long, random string (at least 32 characters)
  3. Different secrets for dev/production
  4. Never share it publicly

  ---
  In Your Code:

  Creating token (login):
  // rtb/backend/controllers/authController.js:39
  const token = jwt.sign({ id: user._id }, process.env.JWT_SECRET, {
    expiresIn: "7d",
  });

  Verifying token (protected routes):
  // rtb/backend/middleware/auth.js:9
  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  req.user = decoded.id;  // Now you know which user made the request

  Frontend stores it:
  // rtb/frontend/src/pages/Login/Login.js:23
  localStorage.setItem("token", data.token);

  ---
  Real-World Analogy:

  Think of JWT_SECRET like the hologram technology on credit cards:

  - JWT = Your credit card with a hologram
  - JWT_SECRET = The secret hologram-making machine only your bank has
  - Signature = The hologram that proves it's real

  Anyone can look at your credit card number (JWT payload is not encrypted),
   but only cards with the correct hologram (signature) are accepted as
  real.

  ---
  Bottom line: JWT_SECRET is what makes your authentication secure. Keep it
  secret, make it random, and never share it!

  2. Backend Changes

  New Authentication Flow:

  A. Google OAuth Controller:

  // rtb/backend/controllers/authController.js

  import { OAuth2Client } from 'google-auth-library';
  import User from '../models/User.js';
  import jwt from 'jsonwebtoken';

  const googleClient = new OAuth2Client(process.env.GOOGLE_CLIENT_ID);

  export const googleLogin = async (req, res) => {
    try {
      const { credential } = req.body; // Google JWT from frontend

      // 1. Verify Google JWT is authentic
      const ticket = await googleClient.verifyIdToken({
        idToken: credential,
        audience: process.env.GOOGLE_CLIENT_ID
      });

      const googleUser = ticket.getPayload();
      // googleUser contains: sub (Google ID), email, name, picture

      // 2. Check if user exists in MongoDB
      let user = await User.findOne({ googleId: googleUser.sub });

      if (!user) {
        // 3. New user - create in MongoDB
        user = await User.create({
          googleId: googleUser.sub,
          email: googleUser.email,
          name: googleUser.name,
          picture: googleUser.picture,
          authProvider: 'google',
          lastLogin: new Date()
        });
      } else {
        // 4. Existing user - update last login
        user.lastLogin = new Date();
        await user.save();
      }

      // 5. Generate YOUR JWT (not Google's)
      const token = jwt.sign(
        {
          id: user._id,
          email: user.email,
          googleId: user.googleId
        },
        process.env.JWT_SECRET,
        { expiresIn: '7d' }
      );

      // 6. Return user data + your JWT
      res.json({
        token,
        user: {
          id: user._id,
          email: user.email,
          name: user.name,
          picture: user.picture
        }
      });

    } catch (err) {
      console.error('Google auth error:', err);
      res.status(401).json({ message: 'Invalid Google token' });
    }
  };

  Creating a Task (with depth calculation)

  // rtb/backend/controllers/taskController.js

  export const createTask = async (req, res) => {
    const { boardId } = req.params;
    const { title, description, parentTask } = req.body;
    const userId = req.user.id;

    // Verify board ownership
    const board = await Board.findOne({
      _id: boardId,
      owner: userId
    });
    if (!board) return res.status(404).json({ message: "Board not found" });

    // Calculate depth
    let depth = 0;
    if (parentTask) {
      const parent = await Task.findById(parentTask);
      if (!parent) return res.status(404).json({ message: "Parent task not 
  found" });
      depth = parent.depth + 1;

      // Optional: enforce max depth
      if (depth > 10) {
        return res.status(400).json({ message: "Max nesting depth reached"
  });
      }
    }

    // Create task
    const task = await Task.create({
      title,
      description,
      board: boardId,
      parentTask: parentTask || null,
      depth,
      createdBy: userId,
      position: 0, // or calculate based on existing tasks
      status: 'todo'
    });

    // Update parent's subtasks array
    if (parentTask) {
      await Task.findByIdAndUpdate(parentTask, {
        $push: { subtasks: task._id }
      });
    }

    res.json({ task });
  };

  Deleting a Task (recursive)

  // rtb/backend/controllers/taskController.js

  export const deleteTask = async (req, res) => {
    const { taskId } = req.params;
    const userId = req.user.id;

    const task = await Task.findById(taskId);
    if (!task) return res.status(404).json({ message: "Task not found" });

    // Verify ownership via board
    const board = await Board.findOne({ _id: task.board, owner: userId });
    if (!board) return res.status(403).json({ message: "Unauthorized" });

    // Recursively delete all subtasks
    const deleteCount = await deleteTaskRecursive(taskId);

    // Remove from parent's subtasks array
    if (task.parentTask) {
      await Task.findByIdAndUpdate(task.parentTask, {
        $pull: { subtasks: taskId }
      });
    }

    res.json({
      message: "Task deleted",
      deletedCount: deleteCount
    });
  };

  async function deleteTaskRecursive(taskId) {
    const task = await Task.findById(taskId);
    if (!task) return 0;

    let count = 1;

    // Delete all subtasks first
    for (const subtaskId of task.subtasks) {
      count += await deleteTaskRecursive(subtaskId);
    }

    // Delete this task
    await Task.findByIdAndDelete(taskId);

    return count;
  }

  Key Design Decisions

  1. Why store both parentTask and subtasks?

  - parentTask: Navigate up the tree
  - subtasks: Navigate down the tree (faster than querying)
  - Denormalized but worth it for performance

  2. Why include board on every Task?

  - Quick filtering: "Get all tasks for board X"
  - Authorization: Check board ownership without traversing tree
  - Analytics: Count tasks per board

  3. Why depth field?

  - Prevent infinite recursion
  - Limit UI nesting (e.g., max 5 levels)
  - Query optimization (e.g., "get only root tasks")
  - Visual indicators in UI

  4. Task deletion strategy:

  - Cascade delete: Delete task ‚Üí delete all subtasks
  - Alternative: Orphan subtasks (move to parent's parent)
  - Your choice depends on UX preferences
